Build & Deploy Pipeline

From source code to production

1
docker_build.sh
Runs npm install && npm run build inside node:22-alpine container. Source mounted via bind mount. Output written to dist/daddy-bear-plumbing/browser/ on host.
2
Angular CLI (ng build)
AOT compilation, tree shaking, code splitting. Produces hashed JS/CSS bundles. Lazy chunks per route (gallery, admin, login, reviews, home).
3
Tailwind CSS (PostCSS)
Scans all .html/.ts files for class usage. Purges unused CSS. Autoprefixer adds vendor prefixes. Final CSS in styles-*.css.
4
TypeScript Compiler
ES2022 target, strict mode, Angular template type-checking. All standalone components, no NgModules.
5
deploy.sh
aws s3 sync dist/.../browser/ s3://kvaking --delete
Syncs build output to site S3 bucket. Removes stale files.
6
CloudFront Invalidation
aws cloudfront create-invalidation --paths "/*"
Clears CDN cache so new content is served immediately.

Local Development

A
docker_serve.sh
Serves pre-built dist/ via nginx:alpine on port 8080. Uses custom nginx.conf with SPA fallback (try_files $uri /index.html).
B
ng serve (optional)
Webpack dev server on port 4200 with hot reload. Requires local Node.js installation.

CloudFront Behaviors

Ordered by specificity — first match wins

Path PatternOrigin (S3 Bucket)Cache PolicyNotes

/reviews-data/reviews.jsonReviewsCachingDisabledAlways fresh — admin writes after changes

403/404 errors on the default behavior serve /index.html (200) for SPA client-side routing.

Route Details

  • /admin is a parent route with children login (public) and "" (guarded)
  • authGuard checks AuthService.isAuthenticated, redirects to /admin/login if false
  • All Angular routes use loadComponent() for lazy loading (separate JS chunks)
  • /admin and /admin/login set <meta name="robots" content="noindex, nofollow">
  • robots.txt disallows /admin — no links to admin from public pages
  • All *.json in gallery bucket are never cached (gallery, meta, services) so updates are instant

Images loaded from /gallery-images/0001_..._tag.jpg

Public Visitor — View Reviews

1
Browser → kvaking.com/reviews
2
Angular loads ReviewsComponent, fetches /reviews-data/reviews.json (not cached)
3
Displays average rating, star ratings, review text, customer name + location

Visitor — Contact Form

1
Visitor fills form: name, email, phone (optional), message
2
Turnstile (invisible) generates verification token in background
3
Browser POSTs to Lambda function URL: form data + Turnstile token + honeypot
4
Lambda checks: honeypot empty? → Turnstile token valid? → required fields present?
5
SES sends email from contactform@domain to business email with reply-to: visitor's email
6
On 429 (throttled): frontend retries 2x with 1s delay before showing error

Admin — Login Flow

1
Admin navigates to /admin
2
authGuard checks: not authenticated → redirect to /admin/login
3
Admin enters username + password → AuthService calls Cognito InitiateAuth
If first login: NEW_PASSWORD_REQUIRED challenge → show new password + confirm fields
4
Cognito returns JWT tokens (ID, Access, Refresh) → saved to sessionStorage
ID/Access tokens expire after 1h (auto-refreshed). Refresh token valid 30 days. Cleared on tab close.
5
Redirect to /admin → guard passes → AdminComponent loads
Page refresh restores session from sessionStorage. Expired tokens silently refreshed.

Admin — Upload Photo

1
Admin sets tag, selects files → filenames auto-generated
Format: 0011_2026-03-29-17-00-00_water-heaters.jpg
2
AuthService calls Cognito Identity Pool → GetCredentialsForIdentity
Returns temp AccessKeyId, SecretAccessKey, SessionToken
3
UploadService signs S3 PUT request with SigV4 using temp creds
Uploads directly to kvaking-gallery/gallery-images/filename.jpg
4
After all uploads: admin lists S3 bucket, writes gallery.json manifest
Public gallery reads this manifest on next page load

Admin — Reorder Photos

1
Admin drags a photo to a new position (drop zones appear between items)
2
Array order updated in memory — no S3 file operations needed
3
Saves reordered gallery.json to S3 (instant, one PUT)

Image Filename Convention

<number>_<datetime>_<tag>.<ext>
0001_2026-02-25-10-30-00_water-heaters.jpg
^^^^  ^^^^^^^^^^^^^^^^^^^  ^^^^^^^^^^^^^
num    Y-m-d-H-M-S          tag slug
  • Number is auto-assigned on upload (not used for ordering — gallery.json controls order)
  • Tag inferred from filename: water-heaters → "Water Heaters"
  • Tag can be overridden per-image by admin (stored in gallery.json)
  • "uncategorized" tag is hidden from display and filter bar
  • Supported extensions: .jpg, .jpeg, .png, .webp

gallery.json Format

[
  "0001_2026-03-28_water-heaters.jpg",
  { "file": "0002_2026-03-28_sink.jpg", "tag": "Kitchen Plumbing" }
]
  • Array order = display order (admin drag-to-reorder)
  • String entry = tag inferred from filename
  • Object entry with tag = custom tag set by admin
  • Backward compatible — old string-only format still works

meta.json Format

{ "hero": "hero-9ae550.png", "og": "og-b3f1c2.jpg" }
  • Stored at gallery-images/meta.json in the gallery bucket
  • Tracks current hero and OG image filenames (hash-based for cache busting)
  • Actual images stored in gallery-images/meta/ subfolder
  • HeroComponent reads this on page load; falls back to hero.jpg if absent
  • CanonicalService reads og for Open Graph image tag

Service Card JSON Format

services-residential.json

[
  { "icon": "water_drop", "title": "Drain Cleaning", "description": "..." },
  { "icon": "plumbing", "title": "Leak Detection", "description": "..." }
]

services-commercial.json

{
  "industries": [
    { "icon": "restaurant", "title": "Restaurants", "description": "..." }
  ],
  "services": [
    { "icon": "build", "title": "Repairs & Maintenance", "description": "" }
  ]
}
  • Stored in gallery bucket at gallery-images/services-*.json
  • Matched by /gallery-images/*.json CachingDisabled pattern
  • Icon names from Material Symbols
  • Pages fall back to hardcoded defaults if JSON doesn't exist
  • Admin "Load Defaults" saves hardcoded cards to S3 for editing

Security Notes

  • All 3 S3 buckets fully private (public access blocked)
  • CloudFront OAC is the only way to read bucket contents
  • Cognito: no self-signup, admin creates users via CLI
  • Admin pages: noindex meta tag + robots.txt disallow + no public links
  • Cognito IAM role scoped to gallery-images/* and reviews-data/* only
  • Terraform state encrypted at rest (SSE-S3)
  • WAF auto-managed by CloudFront security bundle

Fresh Setup Guide

Complete walkthrough from zero — no AWS account, no domain, no infrastructure

Phase 1: Accounts & Tools

1
AWS Account
Go to aws.amazon.com → Create account. Requires email, credit card, phone verification. Select "Personal" or "Business" tier. Free tier covers everything needed.
2
IAM User
AWS Console → IAM → Users → Create user. Attach AdministratorAccess policy (for setup; scope down later). Create access key (CLI access) → save Access Key ID + Secret Key.
3
Domain Registration
AWS Console → Route 53 → Register domain. Pick a domain name. AWS creates the hosted zone automatically. ~$12/year for .com.
4
Cloudflare Account (for Turnstile)
Go to dash.cloudflare.com → Create free account. Application Security → Turnstile → Add site → enter your domain → Widget type: Invisible. Save the Site Key and Secret Key.
5
Install Tools (on your machine)
• Docker — docker.com → Download Docker Desktop (or install Docker Engine on Linux)
• AWS CLI — aws.amazon.com/cli → Download installer → run aws configure → enter Access Key ID, Secret Key, region (e.g. eu-central-1), output format (json)
• Terraform — terraform.io/downloads → Download binary → add to PATH → verify with terraform --version
• Git — git-scm.com (if not already installed)

Phase 2: Repository Setup

6
Clone the repository
git clone <repo-url> && cd daddybearplumbing
7
Configure environment
Edit src/environments/environment.ts:
• phone / phoneDisplay — business phone number (with +1 country code)
• address — business address (line1, city, state, zip)
• AWS values will be filled in after Terraform apply (step 12)
8
Test the build locally
./docker_build.sh — should produce dist/ with no errors
./docker_serve.sh — open http://localhost:8080 to verify

Phase 3: Infrastructure

9
Create Terraform state storage
cd infrastructure
./bootstrap.sh <project-name> <region>
e.g. ./bootstrap.sh daddybear eu-central-1
Creates S3 bucket + DynamoDB table for state locking.
10
Update Terraform backend
Edit the backend "s3" block in main.tf with the bucket/table names from bootstrap output.
11
Configure Terraform variables
cp terraform.tfvars.example terraform.tfvars
Edit terraform.tfvars:
• domain — your registered domain (e.g. daddybearplumbing.com)
• bucket_name — unique S3 bucket name (e.g. daddybearplumbing-site)
• project — short name for resource naming (e.g. daddybear)
• region — AWS region (e.g. eu-central-1)
• contact_email — where contact form emails go (e.g. info+contactform@yourdomain.com)
• turnstile_secret — Secret Key from Cloudflare Turnstile dashboard
12
Handle import.tf
Fully fresh setup (no existing resources): delete import.tf
Existing hosted zone / domain in AWS: copy import.tf.example to import.tf, fill in the hosted zone ID:
aws route53 list-hosted-zones --query "HostedZones[?Name=='yourdomain.com.'].Id" --output text
Uncomment any other pre-existing resources (ACM cert, S3 bucket, CloudFront). After successful terraform apply, you can delete import.tf.
13
Initialize and apply Terraform
terraform init
terraform plan — review all resources to be created
terraform apply — creates: S3 buckets (x3), CloudFront, Route53 records, ACM certificate, Cognito, SES, Lambda
Wait for ACM DNS validation (1-5 minutes).

Phase 4: Angular Configuration

14
Get Terraform outputs
terraform output cognito_user_pool_id
terraform output cognito_client_id
terraform output cognito_identity_pool_id
terraform output contact_form_url
15
Update environment.ts
Paste the Terraform outputs + Turnstile site key into src/environments/environment.ts:
• aws.userPoolId
• aws.cognitoClientId
• aws.identityPoolId
• aws.bucketName / galleryBucket / reviewsBucket (match your terraform.tfvars)
• contactFormUrl
• turnstileSiteKey

Phase 5: Deploy

16
Update deploy.sh
Edit deploy.sh with your S3 bucket name and CloudFront distribution ID:
terraform output s3_bucket_name
terraform output cloudfront_distribution_id
17
Build and deploy
cd ..
./docker_build.sh
./deploy.sh
18
Verify
Open https://yourdomain.com — should show the site with HTTPS.
If DNS hasn't propagated yet, wait a few minutes.

Phase 6: Admin & Email Setup

19
Create admin user
aws cognito-idp admin-create-user --user-pool-id <POOL_ID> --username your@email.com --temporary-password TempPass123
Go to /admin/login → sign in → set new password.
20
SES: Verify receiving email (sandbox)
aws ses verify-email-identity --email-address your@email.com --region <region>
Click the confirmation link in your inbox.
21
SES: Request production access (when ready)
AWS Console → SES → Account dashboard → Request production access.
Mail type: Transactional. Use case: "Contact form, ~5-10 emails/day."
Approved within 24 hours. Until then, only verified emails can receive.

Phase 7: Content

22
Upload gallery photos
Go to /admin → set tag → select photos → upload.
Photos appear on /gallery immediately.
23
Add reviews
Go to /admin → Reviews tab → Add Review → fill in customer name, rating, text, location, date.
24
Test contact form
Go to /contact → fill in form → submit → check your email inbox.

Optional: Google Workspace Email

A
Set up Google Workspace
Go to workspace.google.com → sign up ($6/month for 1 user). Add your domain. Follow DNS verification steps (MX records in Route53).
B
Use plus addressing for contact form
Set contact_email = "info+contactform@yourdomain.com" in terraform.tfvars. Emails arrive in info@ inbox. Set up a Gmail filter to auto-label them.
